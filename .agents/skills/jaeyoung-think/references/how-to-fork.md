# 자기 버전 만들기 — person-think 스킬 제작 가이드

> 이 스킬(jaeyoung-think)은 한 사람의 사고 방식을 외부화한 사례다. 다른 사람도 같은 방법으로 자기 버전을 만들 수 있다. 이 문서는 그 절차를 정리한다.

---

## 이 가이드는 누구를 위한 것인가

- 자신의 사고 방식을 에이전트에 주입하고 싶은 IT 실무자
- 팀에서 특정 시니어의 판단 패턴을 팀 공용 스킬로 만들고 싶은 엔지니어링 리드
- "자신을 외부화한 에이전트" 실험에 관심 있는 AI 엔지니어·연구자
- 개인화된 판단 하네스를 설계하려는 에이전트 시스템 설계자

---

## 사전 조건

1. **충분한 판단 데이터** — 최소 **30개 세션 분량의 대화 로그**가 필요하다. 세션이 부족하면 모드 분포 통계가 노이즈에 묻힌다. 재영의 경우 116개 세션으로 분포가 안정됐다.
2. **요약 포맷** — 각 세션을 events 배열로 요약할 수 있어야 한다. 각 event는 `kind`(discovery/decision/direction/question)와 `content`(한 문장)를 가진다.
3. **작업 타입 분류** — 세션마다 어떤 작업 타입이었는지 태그를 붙일 수 있어야 타입 × 모드 매핑이 가능하다.

---

## 제작 절차 (8 단계)

### 1. 데이터 수집

- AI 에이전트와의 대화 로그, 판단이 담긴 문서, 회고 노트, 설계 기록
- 최소 30 세션, 가급적 100+ 세션
- 같은 작업 타입에 편중되지 않게. 다양한 영역에서 판단 샘플이 필요하다

### 2. 세션별 요약 작성

각 세션을 다음 구조로 압축한다.

```json
{
  "id": "s1",
  "date": "YYYY-MM-DD",
  "title": "한 줄 주제",
  "summary": "세션에서 실제로 한 작업 2~3 문장",
  "events": [
    { "kind": "discovery", "content": "..." },
    { "kind": "decision", "content": "..." },
    { "kind": "direction", "content": "..." },
    { "kind": "question", "content": "..." }
  ],
  "work_types": ["system-design", "documentation"]
}
```

**kind 라벨링 정의**

- `discovery`: 새 사실·현상을 발견한 진술 ("...를 확인했다", "...를 발견했다")
- `decision`: 구체 행동을 확정한 진술 ("...로 결정했다", "...한다")
- `direction`: 방향 재설정 / 원칙 진술 ("...의 방향을 잡았다", "...의 원칙을 세웠다")
- `question`: 열린 질문 ("...는 무엇인가", "...가 맞는가")

**주의**: LLM으로 kind를 자동 라벨링하려면 프롬프트에 위 4가지 정의와 예시를 명시해야 한다. 모호하면 전부 `decision`으로 쏠린다. 수동 라벨링을 10~20개만 해서 LLM 출력과 비교해 보정하는 것을 권장한다.

### 3. 모드 분류기 실행

아래 규칙으로 각 세션을 모드에 배정한다.

- **immediate**: `discovery → decision` 즉시 페어가 하나라도 있음
- **continuous**: `decision → decision` 연속이 하나라도 있음
- **self_review**: `decision → question` 연속이 하나라도 있음
- **exploratory**: 첫 event 또는 마지막 event가 `question`
- **cascading**: 위 4개에 해당 안 하고, 첫 event가 `decision`이며 `{decision, discovery, direction}`이 모두 등장
- **review**: `decision`이 하나도 없음
- **other**: 위 어느 규칙도 안 맞음 (여기가 남으면 규칙 보완 필요)

우선순위: primary 4개(immediate/continuous/self_review/exploratory) > cascading > review > other.

### 4. 분포 확인

모드별 세션 수를 센다. 예상 분포는 사람마다 다르다:

- 답을 알고 시작하는 빠른 결정형: immediate 60%+
- 묶음 작업을 선호하는 체계형: continuous 50%+
- 새 영역을 자주 여는 탐색형: exploratory 20%+
- 자기 의심이 많은 완성형: self_review 15%+

**전체 cover율이 90% 미만**이면 데이터 부족이거나 kind 라벨링 품질이 낮다. 다시 돌아가 요약을 점검한다.

### 5. 작업 타입 × 모드 lift 매트릭스

작업 타입별로 모드 분포를 계산한다. 전체 분포 대비 **lift = 타입 내 모드 비율 ÷ 전체 모드 비율** 을 계산한다.

- lift ≥ 1.5 = 과대표집 (secondary signature 후보)
- lift ≤ 0.7 = 과소표집

타입마다 **1차 모드** (절대 빈도 최대)와 **2차 시그니처** (lift 높은 보조 모드)를 기록한다. 이게 `work-types.md`의 원천이다.

### 6. 원칙 문서 압축

자신의 작업·협업 원칙이 문서로 존재한다면 그걸 10개 안팎의 원칙으로 압축해 `references/principles.md`에 넣는다. 특히 다음을 포함:

- **부정 신호** (거부 패턴) — "나는 이런 옵션을 항상 걷어낸다"
- **선호 어휘** (구체 동사)
- **회피 어휘** (모호 동사)
- **표현 규범** (말투, 형식)
- **각 원칙이 자란 자리** — 이 원칙을 불러낸 실패 또는 교훈의 장면 2~3 문장

**가장 중요한 부분은 거부 패턴이다.** 사람의 판단 고유성은 "무엇을 선택하는가"보다 "무엇을 거부하는가"에서 더 강하게 드러난다. 거부 없는 스킬은 일반 LLM과 구별이 어렵다.

**원칙마다 짧은 스토리를 붙이는 이유** — 외부 독자가 원칙을 추상이 아닌 사람의 경험으로 흡수하게 하기 위해서다. 원칙만 나열하면 framework처럼 읽히고, 스토리가 붙으면 판단 도구로 살아난다.

### 7. SKILL.md 작성

- `name`: `{your-name}-think` 권장 — 재사용성보다 정체성을 강조
- `description`: "범용 프레임워크가 아니다. 특정인의 사고 방식을 외부화한 사례다"를 명시
- 두 모드로 분리: Mode A (다른 에이전트가 판단을 빌릴 때, 구조화 yaml 출력) + Mode B (사람과 자연어로 논의할 때)
- 대화 주제 경계 (Core / Adjacent / Out of scope)
- 안전 경계 (borrowed 표지, 시점 편향, 중요 결정 경고, 단정 금지, 경계 주제 후퇴)
- 한계를 솔직히 드러냄

jaeyoung-think의 `SKILL.md`를 템플릿으로 쓰면 된다. 이름과 인명 관련 부분, 도메인 예시만 자기 것으로 교체하면 된다.

### 8. 검증 루프

**LLM judge는 쓰지 마라.** 실험에서 noisy가 확인됐다 — 같은 입력을 재실행하면 점수가 8/8에서 4/8로 흔들린다. **사람 평가만 ground truth다.**

- **루프 1**: 스킬을 로드한 에이전트의 작업 결과를 본인이 봤을 때 수정이 적게 나는가 측정
- **루프 2**: 소규모 held-out (N=3~5). 본인의 과거 판단 상황을 스킬 없는 에이전트와 스킬 로드한 에이전트에 주고, 본인이 A/B 평가
- **평가 라벨은 보존한다** — N이 쌓이면 미래에 자동 검증의 학습 데이터가 된다

## 자주 나오는 함정

### 함정 1 — 결정 한 줄 예측에 매달림

"이 상황에서 그 사람이 무슨 결정을 할지 한 줄로 맞히기"는 본질적으로 어렵다. 한 시점에서 합리적 선택이 무수하고, 같은 사람도 매번 같은 선택을 하지 않기 때문이다. 결정이 아니라 **사고 흐름**을 빌려주는 쪽으로 프레이밍한다.

### 함정 2 — LLM judge로 "사람다움" 채점

LLM judge는 형식만 보고 noisy하다. 사람 평가만 믿는다. N이 작아도 괜찮다.

### 함정 3 — 거부 패턴 누락

"무엇을 선택할까"에만 집중하고 "무엇을 거부할까"를 놓치면 스킬 출력이 일반적이 된다. 모든 판단에 거부 후보를 최소 1개 포함시키는 것을 규칙으로 만든다.

### 함정 4 — 모호 동사 방치

"최적화", "개선", "강화" 같은 동사는 어떤 사람의 스타일도 포착하지 못한다. 구체 동사로 강제 변환한다.

### 함정 5 — 과설계

MVP는 단일 인스턴스다. 프레임워크화는 N=2(두 명의 person-think가 나왔을 때) 이후로 미룬다. 재사용 가능한 추상을 만들기 전에 자기 버전 하나를 끝까지 가져가본다.

### 함정 6 — 내부 용어 의존

자기 프로젝트의 고유 명칭·개념을 SKILL.md에 그대로 넣으면 외부 독자가 읽을 수 없다. 프로젝트 이름, 파일 경로, 내부 전용 용어는 전부 외부어로 치환한다. "우리 팀의 X 시스템" 대신 "복잡한 소프트웨어 시스템" 식으로.

---

## 외부화의 의미

이 작업은 본인을 복사하는 것이 아니다. 본인의 **사고 방식을 명시적 자산으로 외부화**해서 미래의 자신, 동료, 다른 에이전트가 그 방식을 빌려 쓸 수 있게 하는 것이다.

"내가 없어도 내 판단이 붙어 있게 하는 하네스"를 만든다고 생각하면 방향이 맞다. 또는 이렇게 생각할 수도 있다 — 자신이 팀의 누군가에게 멘토링할 때 말로 전하던 판단 방식을, 24시간 접근 가능한 형태로 명시화하는 것.

## 질문·피드백

이 가이드를 따라 자기 버전을 만들어 본 뒤 발견한 개선점이 있다면 jaeyoung-think 저장소에 이슈로 공유해 달라. 특히 다음 주제는 v2에서 같이 풀어야 할 공통 숙제다:

- **N이 작은 상황에서의 신뢰도 보정**
- **시간 편향 처리** — 사람의 판단 스타일이 변할 때
- **원본 대화 vs 요약 trade-off** — 요약 기반의 누락 신호를 어떻게 보완할지
- **프레임워크화 시점** — 언제 person-think들을 공통 골격으로 묶을지 (N=2 이후의 실제 경험 필요)
