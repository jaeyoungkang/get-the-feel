# product-weaver — 4원리 SSOT

product-weaver는 LLM 에이전트로 AI 제품을 만들 때 의도가 코드까지 왜곡 없이 도달하도록 강제하는 최소 골격이다. 이 문서는 그 골격의 정본이다.

## 출발점

LLM 에이전트의 불확실성 — 환각, 자신 있는 오답, 과설계, scope creep — 은 모두 한 가지 패턴으로 환원된다: **의도 이탈**. 인간이 처음 잡은 의도가 종단(Experience → Moment → Promise → Code)과 횡단(Aspect)으로 짜여가는 동안 어디선가 끊긴다.

product-weaver는 그 끊김을 막는 4원리와, 4원리 자체를 시간 축에서 다듬는 메타 프로세스 하나만 들고 있다. 더 줄이면 의도 이탈이 새고, 더 늘리면 도메인이 묻혀 들어와 비대해진다. 이 4원리는 **공리** — 깎으면 안 되는 것. 메타 프로세스는 **루프** — 멈추면 안 되는 것.

도메인 인스턴스는 `references/`에서 참조한다.

---

## 원리 1 — Intent-Evidence Chain (의도-증거 사슬)

**정의**
의도(Intent)는 명시되어 잠기고, 잠긴 의도와 결과의 갭은 append-only 원장(Ledger)에 기록되며, 원장은 기계가 검증(Verdict)한다. 이 세 단계는 한 사슬이며 끊기지 않는다.

```
Intent Lock → Gap Ledger → Mechanical Verdict
```

**막는 LLM 패턴**
- 추상 의도만 주면 LLM이 자기 해석으로 빈틈을 채운다
- "다 됐다"고 거짓 보고한다
- 원장도 LLM이 거짓으로 채울 수 있다

**인스턴스화 가이드**
- Intent Lock 인스턴스: 제품 정체성 문서, 약속, 샘플 artifact, 엔지니어링 결정 문서. 형식은 도메인이 결정.
- Gap Ledger 인스턴스: 검증 원장, 변경 로그, 운영 피드백 ledger. 형식은 도메인이 결정. **append-only**는 공리.
- Mechanical Verdict 인스턴스: 정합성 검사 CLI, CI gate, 배선 검증 스크립트. 구체는 도메인이 결정. **unknown/not-met은 release blocking**은 공리.

## 원리 2 — Preflight Gate (선행 게이트)

**정의**
선행 의도 잠금 없이 다음 단계로 진행할 수 없다. 진행 직전에 기계가 선행 산출물의 존재와 형태를 확인하고, 빠지면 차단한다.

**막는 LLM 패턴**
- 의도 없이 코드부터 쓴다
- "일단 시작하면 의도는 따라온다"는 환상
- 묵시적 생략 — "이번엔 sample 안 만들어도 될 것 같다"

**인스턴스화 가이드**
- Gate의 위치는 도메인이 결정 — bootstrap 직전, release 직전, propagate 직전 등
- Gate의 확인 항목은 원리 1의 Intent Lock 산출물
- **묵시적 생략 금지** — 생략은 항상 명시적 사유와 함께
- Human approval이 필요한 Gate와 기계 점검만으로 통과하는 Gate를 구분

---

## 원리 3 — Atomic Step (단위 잠금)

**정의**
한 번의 진행 단위는 작게 잠근다. 한 turn = 한 작업 항목 = 한 commit. 한 단위 안에서 여러 의도를 섞지 않는다.

**막는 LLM 패턴**
- 한 번에 여러 일을 섞어 처리하면서 일부를 누락
- 실패한 단위 안에서 연쇄 수정으로 빠져나가려 함
- 큰 단위가 어디서 무너졌는지 추적 불가

**인스턴스화 가이드**
- 작업 큐는 체크박스 리스트로 명시
- 한 단위가 한 turn 안에 닫히지 못하면 분해
- 단위 안에서 검증이 실패하면 변경을 보존한 채 원인과 다음 최소 수정안을 기록하고 단위 종료
- **묶기 금지** — 한 commit에 여러 단위 묶지 않는다

---

## 원리 4 — Refuse First (거부 우선) — 메타 원리

**정의**
의도 규율은 "무엇을 할까"보다 "무엇을 안 할까"를 먼저 잠근다. 진행 단위마다, 산출물마다, 스킬마다, 도구 추가마다 **Never** 와 **Ask first** 를 먼저 명시한다.

**왜 메타인가**
원리 1·2·3은 "잘못된 진행"을 막는 가드다. 원리 4는 **진행 자체의 방향**을 막는 가드라서 한 단 위에 있다. 1·2·3을 운영하는 태도가 4다.

**막는 LLM 패턴**
- 기본값 "yes, and" — 사용자 의도에 모두 동의·확장
- 거절 비용 과소평가, 수용 비용 과소평가
- "더 좋게 만들 수 있다"는 환상으로 과설계
- "사용자가 원하는 것 같다"는 추정으로 scope creep

**인스턴스화 가이드**
- 모든 AGENTS.md / 진행 문서는 **Never → Ask first → Always** 순으로 잠근다
- 새 도구·스킬·원리·문서 추가는 "거부할 사유"를 먼저 검토
- 판단 작업(범위, 기술 선택, 추상화 결정)은 "거부할 것"을 먼저 명시한 뒤 "추가할 것"을 결정

---

## 4원리의 관계

```
                   ┌─── Refuse First (메타) ───┐
                   │                            │
                   ▼                            │
     Intent Lock ──→ Gap Ledger ──→ Mechanical Verdict
                                                ▲
                                                │
                       Preflight Gate ──────────┘
                                                ▲
                                                │
                       Atomic Step ─────────────┘
```

- **Intent-Evidence Chain**: 종단축 — 의도가 코드까지 도달하는 본 경로
- **Preflight Gate**: 사슬 시작 전 잠금 — 의도 없이 사슬에 진입 차단
- **Atomic Step**: 사슬을 짜는 한 stitch의 크기 — 한 번에 한 매듭
- **Refuse First**: 사슬 짜기의 방향 결정 — 무엇을 짜지 않을지 먼저

Intent Weaving 어휘로 보면 4원리는 **weave를 잠그는 4축**이다.

---

## 메타 프로세스 — Refinement Loop (다듬기 루프)

4원리는 정적 명세다. 정적 명세는 시간이 지나면서 도메인 학습이 누적되고 새로운 LLM 이탈 패턴이 식별되면서 부정확해진다. 따라서 **4원리 자체를 평가·갱신하는 루프**가 같이 있어야 product-weaver가 살아있다.

원리 1~4는 weave를 잠그는 축이고, Refinement Loop는 **그 축들을 시간 축에서 다듬는 루프**다. 이 루프 없이는 베이스가 박제된다.

**막는 LLM 패턴**
- "한 번 잠근 규율은 변하지 않는다"는 환상
- 운영 결과를 원리에 되먹이지 않고 코드만 계속 짜는 패턴
- 새 의도 이탈 패턴을 식별하고도 기존 4원리에 무리하게 끼워 맞추기
- 도구·스킬·문서가 한 방향으로만 늘어나면서 비대해지는 패턴

**인스턴스화 가이드**

이 루프의 입력·출력·주기는 다음을 만족해야 한다.

- **입력**
  - Intent-Evidence Chain의 운영 결과 (증거 누적, 검증 실패 패턴)
  - Preflight Gate를 통과한 뒤 발생한 의도 이탈
  - Atomic Step에서 무너진 단위 기록
  - Refuse First로 거부된 변경의 사후 평가
- **출력**
  - 원리의 **인스턴스화 가이드** 갱신 (원리 본문은 거의 변하지 않는다 — 인스턴스화 가이드가 진화의 본 자리)
  - 새 LLM 이탈 패턴 식별 시 원리 추가 검토 (아래 셀프 게이트)
  - 도메인 인스턴스의 폐기·승격 결정과 그 사유
- **주기**
  - 도메인이 결정하되 **묵시적 생략 금지** — 주기는 명시되어야 한다
  - 주기마다 운영 피드백 ledger와 의도 판단 기록을 검토

**비대화 방지 셀프 게이트** — 새 원리·도구·문서·스킬 추가 검토 시 다음 질문을 통과해야 한다.

1. 이것은 LLM 불확실성 규제의 **동형 원리** 인가, 특정 도메인의 인스턴스인가?
2. 4원리 중 하나의 인스턴스로 충분히 표현되는가? 그렇다면 베이스에 올리지 않고 도메인에 둔다.
3. 기존 4원리로 표현할 수 없는 LLM 이탈 패턴이 새로 식별되었는가? 그 패턴은 어떤 운영 증거로 뒷받침되는가?
4. 추가했을 때 베이스의 무게가 늘어나는 만큼, 어떤 의도 이탈 패턴이 막히는가?

**1·2를 통과하지 못하면 거부.**

**자기 적용** — Refinement Loop 자신도 4원리를 따른다. 운영 결과는 의도-증거 사슬에 기록되고, 정기 회고는 Preflight를 거쳐 시작되며, 한 회고 = 한 단위 = 한 commit으로 잠기고, 변경은 거부 우선으로 검토된다.
