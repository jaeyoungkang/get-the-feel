# 명확한 경계 (Clear Boundaries)

> 에이전틱 엔지니어링의 핵심은 "모든 작업 단위의 경계를 명확히 하는 것"이다.
> 전체 맥락은 `principles.md` 참조.

## 계산과 효과를 분리한다

모든 층에서 "순수한 계산"(입력→출력이 예측 가능, 부수효과 없음)을 최대화하고, "효과"(네트워크, 상태 변경, LLM 호출)를 가장자리에 격리한다.

| 층 | 계산 (순수, 테스트 가능) | 효과 (비순수, 격리) |
|---|---|---|
| **코드** | 데이터 변환, 필터, 매핑 | API 호출, 상태 변경, DOM 조작 |
| **비즈니스 로직** | 정책 판단, 점수 계산, 약속 정리 | LLM 호출, DB 쿼리 |
| **프롬프트/응답** | 프롬프트 조립, 응답 파싱 | LLM 추론 |

**왜 이것이 핵심인가:**
- 순수한 계산은 LLM이 정확히 이해하고 안전하게 수정할 수 있다
- 테스트가 자동화된 리뷰어 역할을 한다
- 패턴이 일관적이면 LLM이 같은 패턴으로 쓴다 — 품질이 전파된다
- 저품질 코드는 토큰을 낭비한다 — 코드 품질이 곧 비용 효율

## 경계를 강제하는 도구

각 도구가 명확히 하는 경계의 종류가 다르다:

| 도구 | 명확히 하는 경계 | 질문 |
|------|----------------|------|
| Formatter (Prettier) | 표현의 경계 | 이 코드는 어떤 모양이어야 하는가 |
| Linter (ESLint) | 규칙의 경계 | 이 코드는 어떤 패턴을 따라야 하는가 |
| jscpd | 책임의 경계 | 이 로직은 한 곳에만 있어야 하는가 |
| Test (vitest/jest) | 행동의 경계 | 이 함수는 무엇을 해야 하는가 |
| Coverage | 검증의 경계 | 검증되지 않은 코드가 어디인가 |
| TypeScript | 데이터의 경계 | 이 값은 어떤 형태여야 하는가 |
| dependency-cruiser | 의존의 경계 | 이 모듈은 누구를 알아도 되는가 |
| knip | 존재의 경계 | 이 코드는 아직 필요한가 |
| Concern Check | 관심사의 경계 | 이 값은 한 곳에서 선언되고, 사용 지점에서 참조되는가 |

공통 원리: **암묵적 경계를 명시적으로 만들고, 자동으로 강제한다.** 사람끼리는 암묵적 합의가 통하지만, 에이전트는 암묵적 합의를 모른다.

## 관심사 중앙화 (Cross-Cutting Concern Centralization)

여러 파일에 걸쳐 반복되는 값(디자인 토큰, 에러 메시지, API 경로, UI 문자열 등)을 한 곳에서 선언하고, 사용 지점에서 참조하게 만들고, 직접 값 사용을 기계적으로 차단하는 패턴이다.

### 선언-매핑-강제 3단

| 단 | 역할 | 빠지면 |
|---|------|--------|
| **선언** (Declaration) | 값의 단일 원본을 한 파일에 정의 | 뭘 써야 하는지 모름 |
| **매핑** (Mapping) | 선언된 값을 사용 지점에 연결하는 메커니즘 | 정의했지만 아무도 안 씀 |
| **강제** (Enforcement) | 우회(하드코딩)를 기계적으로 탐지·차단 | 에이전트가 직접 값을 써서 일관성 붕괴 |

세 단이 모두 갖춰져야 관심사가 "중앙화"된 것이다.

### 에이전트 관점

프롬프트로 "토큰을 써라"고 부탁하는 것은 불안정하다. 하드코딩을 CI가 거부하면 에이전트는 토큰을 쓸 수밖에 없다. 관심사 중앙화의 강제 단계가 곧 품질 게이트다. "프롬프트로 부탁하는 게 아니라 스크립트로 CI에서 자동 차단한다"의 관심사 버전.

### 적용 대상과 강제 규칙

| 관심사 | 선언 | 매핑 | 강제 (검증 규칙) |
|--------|------|------|-----------------|
| 디자인 토큰 | CSS 변수 (`:root`) | `@theme inline` → 유틸리티 | `rounded-\[\d+px\]` 등 하드코딩 패턴 0건 |
| 에러 메시지 | 에러 카탈로그 | `throw new AppError("CODE", {...})` | 인라인 `throw new Error("...")` 탐지 |
| API 경로 | 상수 파일 | `import { API_ROUTES }` | 인라인 `"/api/..."` 문자열 탐지 |
| UI 문자열 | 메시지 파일 (`i18n/messages.ts`) | `t("key", {param})` | JSX 내 하드코딩 문자열 탐지 |

### 판단 기준: 무엇을 중앙화할 것인가

모든 값을 중앙화하면 과설계다. 판단 기준:
1. **파일 수** — 3개 이상의 파일에서 반복되는가
2. **변경 동기** — 한 번에 전부 바꿔야 할 이유가 있는가 (테마, 언어, 환경)
3. **에이전트 일관성** — 에이전트가 새 코드를 쓸 때 올바른 값을 모를 위험이 있는가

세 기준 중 둘 이상 해당하면 중앙화 대상이다.

## 보완 개념

- **불변성** — 데이터의 경계를 명확히 한다 (순수 함수는 함수의 경계)
- **단일 책임** — 변경 이유의 경계를 명확히 한다
- **의존성 주입** — 비순수 부분을 교체 가능하게 해서 순수 함수를 가능하게 한다
- **계약/타입** — 경계를 코드(타입 시스템, Zod 스키마)로 강제한다

## 에이전트가 만드는 코드의 약점과 탈출구 차단

에이전트는 **꼼수를 먼저 시도하고**, **코드 삭제를 잘 하지 않고**, **중복 테스트를 과다 생성**하며, **한 파일에 기능을 계속 쌓는** 경향이 있다. 프롬프트로 부탁하는 게 아니라 **기계적으로 차단**한다.

### 탈출구 차단 규칙 (ESLint + 스크립트)

| 약점 | 구체 규칙 | 목적 |
|---|---|---|
| `@ts-ignore` / `@ts-expect-error` 남발 | `@typescript-eslint/ban-ts-comment`: 최소 10자 설명 필수 | 무설명 탈출 금지 |
| 테스트 스킵 | `describe.skip` / `it.skip` / `test.skip` 금지 (ESLint 커스텀 규칙) | 기능 붕괴 은폐 방지 |
| `eslint-disable` / `any` | ESLint `max-warnings=0` + `any` 경고 | 우회 거부 |
| feature 간 직접 접근 | `no-restricted-imports`: `@/app/features/*/*` 차단 | 경계 침범 |
| 파일 비대화 | `max-lines: 700` | 읽기·이해 붕괴 |
| 함수 비대화 | `max-lines-per-function: ts 220 / tsx 350` | 책임 과적재 |
| 복잡도 폭주 | `sonarjs/cognitive-complexity: 25` | 분기 미로 |
| 죽은 코드 누적 | `knip` (미사용 export 탐지) | 삭제 거부 대응 |
| 중복 테스트 폭주 | 스크립트 `check-test-ratio.mjs`: 테스트/프로덕션 LOC 120% 이하 | 테스트 폭식 차단 |
| 중복 코드 블록 | `jscpd`: 8줄·80토큰 이상 중복 0건 | 복붙 감지 |
| 하드코딩 한글 | 스크립트 `check-hardcoded-korean.mjs`: JSX/JS 내 한글 문자열 차단 | i18n 우회 방지 |
| HTML escape 우회 | 스크립트 `check-escapes.mjs` | XSS 회피 |

### 도구 조합이 중요하다

하나만 강제하면 다른 쪽이 터진다. 예:
- max-lines만 있고 knip이 없으면 → 파일 쪼개지만 죽은 파일이 누적
- test.skip 금지만 있고 커버리지 임계치가 없으면 → 테스트 수는 많지만 실제 로직 미커버
- concern check만 있고 ESLint no-restricted-imports가 없으면 → 값은 중앙화했지만 모듈 경계는 침범

이 표의 규칙들은 **세트로 작동한다**.
