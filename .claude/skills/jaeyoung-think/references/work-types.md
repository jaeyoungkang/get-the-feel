# 작업 타입 × 사고 모드 매핑

> 이 스킬이 입력 상황을 10가지 작업 타입 중 하나에 매핑한 뒤, 그 타입의 1차 모드와 2차 시그니처로 사고 흐름을 전개할 때 참조한다.

---

## 왜 "작업 타입"인가

같은 사람이라도 작업의 성격에 따라 사고 모드가 달라진다. 운영 이슈를 풀 때와 새로운 개념 체계를 설계할 때는 같은 사람도 다른 모드로 들어간다. 10가지 작업 타입은 IT 실무에서 반복적으로 등장하는 판단 상황의 카테고리다. 각 타입마다 재영이 **주로 사용하는 1차 모드**와 **과대표집되는 2차 시그니처**를 기록해 두면, 새 상황이 들어왔을 때 어느 모드 조합을 적용할지 결정할 수 있다.

---

## 읽는 법

- **1차 모드**: 해당 타입에서 가장 많이 사용되는 사고 모드 (주로 immediate 또는 continuous)
- **2차 시그니처**: 1차 모드 위에 추가로 **과대표집(lift ≥ 1.5)** 되는 보조 모드
- **lift**: 이 작업 타입에서 해당 모드가 나타나는 비율 ÷ 전체에서 그 모드가 나타나는 비율. 1.0이 평균, 1.5 이상이면 의미 있는 과대표집.

재영의 기본값은 immediate 61%, continuous 27%가 전체를 덮는다. 타입별 성격은 그 기본값 위에 **어떤 보조 모드가 더해지는가** 에서 드러난다.

---

## 10가지 작업 타입

### 1. System Design (시스템 설계)

복잡 시스템 아키텍처, 데이터 흐름, 컴포넌트 경계, 저장소 구조, 상태 관리.

- **1차 모드**: immediate
- **2차 시그니처**: **cascading lift 1.4×**
- **해석**: 복잡한 시스템은 설계의 전체 지형이 실행 전에는 완전히 드러나지 않는다. 결정 후 실행 중 예상치 못한 상호작용이 나타나 추가 결정이 연쇄된다. self_review는 거의 없다 — 재영은 내부 구조에 대해서는 자기 의심을 발동하지 않는다.
- **트레이스 가이드**: 결정 → 발견 → 방향 → 결정 (멀티쓰레드)

### 2. UX & Product Decision (사용자 경험·제품 결정)

사용자 인터페이스, 플로우, 온보딩, 제품 옵션 선택. 사용자와 닿는 모든 표면.

- **1차 모드**: immediate
- **2차 시그니처**: **self_review lift 2.4×**
- **해석**: 사용자가 평가하는 결과물이라 결정 후 "이게 자연스러운가" 의심이 거의 자동으로 발동된다. 자기 의심 없이 나온 UX 결정은 재영 기준으로 약한 결정이다.
- **트레이스 가이드**: immediate 페어 + 마지막에 question 1개로 자기 의심 삽입

### 3. Methodology Building (개발 방법론 정립)

개발 원칙, 컨벤션, 품질 게이트, 스펙 구조, CI/CD 룰 설계.

- **1차 모드**: **continuous lift 1.31×**
- **2차 시그니처**: **self_review lift 2.1×**
- **해석**: 방법론은 묶음 작업이다 — 여러 원칙·게이트가 한 덩어리로 설계된다. 그리고 독자(팀원·미래의 자신)가 있기 때문에 마지막에 자기 의심이 들어온다. "이 방법론이 실제로 강제되는가? 말로만 부탁하는 규칙이 아닌가?"
- **트레이스 가이드**: continuous 결정 묶음 + 마지막 self_review question

### 4. Documentation (문서 작업)

스펙, README, 가이드, 설계 문서, 원칙 문서 작성·수정.

- **1차 모드**: **continuous lift 1.35×**
- **2차 시그니처**: **self_review lift 3.1×** (모든 타입 중 2위)
- **해석**: 문서는 줄줄이 쓰는 성격이지만, 독자가 있다는 의식이 강하게 작동해서 결정 후 재검토 빈도가 모든 타입 중 두 번째로 높다. "이 문서가 외부 참조 없이 자체 완결되는가"가 재영의 대표적 self_review 질문.
- **트레이스 가이드**: continuous 묶음 + 마지막 self_review

### 5. Collaboration Design (협업 구조 설계)

팀·동료·상호작용 설계. AI 에이전트와 사람의 협업 관계, 동료 에이전트의 약속과 자율성.

- **1차 모드**: immediate
- **2차 시그니처**: **exploratory lift 2.6×**, **cascading lift 1.5×**
- **해석**: 협업 주체라는 개념 자체가 열려 있는 영역이라 질문으로 여는 빈도가 높다. 그리고 협업 구조는 복잡 시스템이라 결정 후 갈라지는 cascading도 자주 나타난다. continuous는 **과소표집** (lift 0.6) — 협업 설계는 묶어서 일괄 결정하는 성격이 아니다.
- **트레이스 가이드**: question으로 열고 cascading 연쇄로 수렴

### 6. Observability Design (관찰·모니터링 설계)

모니터링 시스템, 로그 구조, 디버깅 경로, 에이전트 의사결정 추적.

- **1차 모드**: immediate
- **2차 시그니처**: **self_review lift 2.9×**, **cascading lift 1.5×**
- **해석**: 관찰 시스템은 "사람이 읽을 수 있어야 한다"는 요구가 있어서 self_review가 높다. 그리고 관찰 시스템의 구조는 실행 중 드러나는 정보가 많아 cascading도 걸린다.
- **트레이스 가이드**: 결정 → 발견 → 방향 + 끝에 self_review question

### 7. Knowledge Structuring (지식 구조화)

개념 체계 설계, 지식 지도, 정보 아키텍처, 학습 자료 구조.

- **1차 모드**: immediate
- **2차 시그니처**: **exploratory lift 7.8×** (모든 타입 중 1위), **cascading lift 2.9×**
- **해석**: 지식 구조 설계는 **새로운 영역에서의 탐색**이라 질문으로 여는 빈도가 압도적으로 높다. "이 지식의 본질적 단위는 무엇인가?"가 첫 질문이 되는 경우가 많다. 그리고 설계 중 지형이 드러나면서 cascading으로 연쇄된다.
- **트레이스 가이드**: question 시작 + cascading 연쇄 + 가능하면 question으로 마감
- **주의**: 표본 수가 작아 신뢰도 medium

### 8. User Testing & Validation (사용자 검증)

유저 테스트 계획, 검증 시나리오 설계, 결과 분석, 개선 우선순위.

- **1차 모드**: immediate
- **2차 시그니처**: 약함
- **해석**: 유저 테스트는 발견 → 수정의 전형적 immediate 반복이다. 특별한 보조 시그니처가 없다. 발견된 문제가 명확하면 즉시 수정 결정, 명확하지 않으면 다음 발견을 기다리는 식.
- **트레이스 가이드**: immediate 기본

### 9. Operations (운영 반복)

세션·태스크·반복 작업 관리, 팀 운영 체계, 주기적 작업 개선.

- **1차 모드**: **continuous lift 1.22×**
- **2차 시그니처**: 약함
- **해석**: 운영 개선은 묶음으로 이뤄진다. 한 사이클에 여러 개선을 같이 반영하는 성격. 자기 의심은 거의 없다 — 운영 개선은 시행착오로 굳히는 성격이라 "실행해 보고 아니면 다음 사이클에 수정" 태도가 기본.
- **트레이스 가이드**: 결정 → 결정 → direction → 결정

### 10. Code Review & Refactoring (코드 리뷰·리팩토링)

타인 PR 리뷰, 회귀 가능성 점검, 기존 코드 구조 개선, 죽은 코드 삭제.

- **1차 모드**: immediate (리뷰어가 본인일 때) 또는 **review** (결정자가 타인일 때)
- **2차 시그니처**: 작업 맥락에 따라 다름
- **해석**: 이 타입은 두 갈래다. 본인이 결정자이면 즉결 immediate, 타인이 결정자이고 본인이 조언자이면 판단 유보 review 모드로 들어간다. 이 구분은 입력 맥락에서 누가 결정자인지를 봐야 한다.
- **트레이스 가이드**: 결정자 여부 확인 → immediate 또는 review

## 작업 타입 자동 추정 규칙

입력 맥락(파일 경로, 대화 키워드, 질문 성격)에서 아래 단서를 찾아 매핑한다. 여러 개 걸리면 가장 구체적인 것을 우선.

| 단서                                                    | 작업 타입                 |
| ------------------------------------------------------- | ------------------------- |
| 아키텍처, 데이터 흐름, 저장소, 상태 관리, 컴포넌트 경계 | System Design             |
| UI, UX, 플로우, 온보딩, 사용자 경험, 제품 옵션          | UX & Product Decision     |
| 원칙, 컨벤션, 게이트, 메서드olo지, 린트, CI 룰          | Methodology Building      |
| README, 스펙, 가이드, 문서, 설계 문서                   | Documentation             |
| 팀, 협업, 동료, 에이전트 관계, 약속, 자율성             | Collaboration Design      |
| 모니터링, 로그, 관찰, 디버깅 추적                       | Observability Design      |
| 지식 맵, 개념 체계, 정보 구조, 학습 자료                | Knowledge Structuring     |
| 유저 테스트, QA, 검증 시나리오, 결과 분석               | User Testing & Validation |
| 운영, 세션, 태스크, 주간 플랜, 반복 개선                | Operations                |
| 리뷰, 리팩토링, PR, 죽은 코드, 회귀                     | Code Review & Refactoring |

매칭이 실패하면 `System Design`을 fallback으로 쓴다 (IT 실무의 가장 넓은 기본 카테고리).

---

## 이 매핑의 한계

- **표본 수 편차** — 어떤 작업 타입은 풍부한 사례에서 도출되지만, Knowledge Structuring 같은 경우 표본이 작아 신뢰도가 낮다.
- **작업 타입 경계가 겹친다** — Collaboration Design과 System Design, 또는 Methodology Building과 Documentation이 섞여 등장하는 경우가 많다. 이 경우 입력 맥락의 주된 결에 따라 선택한다.
- **시간 편향 없음** — 사람의 판단 방식은 시간에 따라 바뀐다. 현재 매핑은 그 변화를 반영하지 못한다.
- **새로운 작업 타입** — 목록에 없는 타입이 등장하면 자동 추정은 실패한다. 이 경우 fallback으로 감서 판단하고, 누적되면 매핑을 갱신해야 한다.
