# Goal

Turn a local visualization into a feedback-ready product candidate.

## When To Use

- When a candidate claims customer proof, onboarding, pricing, watchlist, waitlist, or paid readiness.
- Before writing `product_feedback_readiness`, `repeat_use_readiness`, or `pricing_evaluation_readiness`.
- When deciding the next loop after R5/R6-style customer proof or blocker panels.

## Workflow

0. **Skill Load Receipt** — record this skill path and candidate id in `assets/CYCLE_RECORD.md`.
1. **Feedback Capture** — provide a visible action or structured surface for a user to give feedback, choose a need, join a waitlist, or express intent. Static copy is not enough.
2. **Repeat-Use Workflow** — show the user what they would check again on a cadence and why.
3. **Pricing Evaluation** — expose a testable pricing/offer choice without claiming validated demand.
4. **Macro Promise Preservation** — feedback and pricing surfaces must remain subordinate to the macro bottleneck map.
5. **Next Evidence Path** — record what user signal would upgrade the candidate and where it would be stored.

## Output Checks

- feedback_capture_visible: pass / blocked
- repeat_use_workflow_visible: pass / blocked
- pricing_evaluation_visible: pass / blocked
- macro_map_preserved: pass / blocked
- next_evidence_path_defined: pass / blocked

## Never

- Never count external blockers as customer feedback.
- Never count pricing copy as pricing evaluation unless the user can choose or respond.
- Never let onboarding, waitlist, or pricing replace the map.
- Never mark a candidate feedback-ready when the only action is reading text.
