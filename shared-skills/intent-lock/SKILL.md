# Goal

새 의도 — 제품, 약속, 샘플, 엔지니어링 결정 — 가 처음 들어왔을 때, **추상에서 명시로 옮기는 인터뷰·구체화·잠금** 을 거친다.

LLM의 기본 실패 패턴은 추상 의도를 자기 해석으로 채우는 것. 이 스킬은 그 자리를 사람(Human authority)에게 돌려준다.

## When To Use

- 새 의도가 들어왔는데 아직 명시되지 않았다
- 사용자가 막연한 아이디어 단계에서 잠금 산출물로 넘어가야 한다
- 기존 잠금 산출물에 큰 빈틈이 식별되었고, 다시 인터뷰가 필요하다

## Workflow

0. **Skill Load Receipt** — 이 파일을 실제로 읽었음을 산출물 또는 진행 메시지에 남긴다.
   - `skill: shared-skills/intent-lock/SKILL.md`
   - `loaded_at_step`
   - 이번 의도 잠금에 적용할 required gates
   - 잠금 전 차단 조건(`blocked_until`)
   Receipt 없이 인터뷰·draft·코드 작성으로 넘어가면 Preflight 실패.

1. **Refuse First 진입** — "거부할 사유"부터 검토.
   - 이 의도를 잠그지 않아도 되는가? (의도가 베이스 4원리와 충돌하지는 않는가)
   - 잠그더라도 베이스에 두지 말고 도메인에 두면 되는 것 아닌가?
   - "거부할 사유"가 없을 때에만 다음 단계로.

2. **현재 단계 선언** — `discover` / `draft` / `approve` 중 하나로 명시. 단계 없이 진행하지 않는다.

3. **소단위 인터뷰** — 한 라운드에 1~3개의 좁은 질문. 큰 설문지 한 번에 던지지 않는다.
   - 의도의 경계(무엇을 포함·제외하는가)
   - 의도의 운영 형태(잠그는 결과물이 일회성 아티팩트인가, 지속 운영될 시스템인가) — 답에 따라 후속 잠금 산출물 형태가 달라진다
   - 의도의 차원(이번 사이클에서 잠글 *사이클 의도*인가, 사이클들을 통해 자라날 *메타 의도*인가) — 메타 의도는 한 번에 잠기지 않는다. Spiral Loop 적용 자리 (`refinement-loop/SKILL.md` 참조)
   - **의도의 코드 도달 단위** — 이 의도가 결국 *어떤 코드 단위·사용자 표면·데이터 항목·기계 검증 가능한 산출물*로 떨어지는가? 한 줄로 답해본다. 답이 안 나오면 의도가 너무 추상이거나 Intent-Evidence Chain(의도가 코드까지 도달)과 충돌하는 종류. Spiral Loop 운영 중이면 *언제까지 어느 사이클에 도달*인지도 함께 명시. (도메인의 종착이 코드가 아닌 경우 — 정책·교육 콘텐츠·문서 등 — *도메인 종착 산출물*로 치환한다.)
   - **자산 기여 (WHO·WHY 잠금 포함)** — 이 의도가 도메인의 *자산 체계*에 무엇을 기여하는가? 자산 카테고리 1개 명시 + 기여 형태 한 줄. 기여 1조각은 *누가(WHO)·왜(WHY)*의 한 줄 갱신을 반드시 포함한다 — *코드가 누구의 무엇을 위한지 휘발되는* 함정 차단.
     **자산 카테고리는 도메인이 스스로 발견한다 — 베이스가 예시 목록을 강제하지 않는다.** 첫 잠금 또는 자산 체계 회고 시점에 다음 질문을 던진다:
     - 이 제품이 *제대로 작동하려면* 코드 외에 *무엇이 함께 자라야* 하는가?
     - 그것들은 *평행하게 진행*되는가, *순차적*인가? (평행이면 별 trunk)
     - 각 자라남이 *검증되는 자리*는 어디인가?
     - 이미 채택한 카테고리가 *고민의 결과*인가, *예시 모방*인가? 후자면 폐기 후보.
     발견된 카테고리는 도메인 AGENTS.md에 명시 + 채택 사유 박는다. 다른 도메인의 카테고리 목록을 *수용*하지 않는다.
     **제품 도메인 Asset Map Gate** — 새 제품을 만들거나 Spiral Loop를 시작하는 경우, 첫 사이클 전에 도메인 Asset Map을 둔다. 이름과 파일 구조는 도메인이 정하지만, 아래 capability가 자산으로 존재하거나 명시적으로 거부되어야 한다:
     - 제품 계약/사용자 약속
     - 제품별 스킬/플레이북
     - 엔지니어링 규약 또는 구현 패턴
     - 리서치·데이터 구성·출처 운영
     - 화면 디자인·시각화·UX 판단
     - 비즈니스 로직·가격·판매 가능성 판단
     - 프로세스 개선/서브에이전트 모니터링 회수
     각 capability마다 `asset_path`, `why_it_must_grow`, `how_to_update`, `mechanical_or_review_gate`, `last_cycle_contribution`을 적는다. 스킬/플레이북 capability는 추가로 `when_to_use`, `required_receipt`, `mechanical_enforcement`를 적는다. 비어 있으면 *자산 없는 제품 사이클*로 보고 시작하지 않는다.
   - **제품 계약 발전** — 이 사이클 의도가 *사용자에게 한 약속*을 어떻게 발전시키는가? 추가·수정·검증 중 하나 명시. 한 줄 답: "이 사이클로 [약속 X]가 [추가됨/구체화됨/검증됨/반증됨]". 답할 수 없으면 *사용자 약속을 진전시키지 않는 사이클* — 자산 기여만으론 충분 안 함, 의도 재정의 또는 폐기. **제품 계약 자산** (도메인이 자기 형식으로 정의: Hypothesis Stitch · user story · promise · contract 등)이 *모든 제품 도메인에 필수*이며 매 사이클의 배움이 그곳에 회수된다. 자산 기여(공급 측 — 코드·design·data)와 짝으로 제품 계약 발전(수요 측 — 사용자 약속). 둘 다 명시되어야 사이클 의도 잠금 완성.
   - 거부 신호(어떤 형태가 되면 안 되는가) — Refuse First를 인터뷰에 내장
   - 검증 방법(어떻게 잠긴 의도와 결과의 갭을 확인할 것인가)

   **위임 모드 (Delegation Mode)** — 사용자가 명시적으로 의사결정을 위임할 때 ("주요 의사결정은 니가 알아서 선택하라" 같은 발화). 이 경우:
   - 인터뷰 라운드를 압축한다 (한 라운드 또는 0 라운드)
   - 에이전트가 합리적 기본값으로 일방 결정 + **모두 미승인 표기**
   - 미승인 항목은 사후 승인 큐(`fix_plan.md`)에 항목으로 등록
   - 위임 모드 사용 사실을 잠금 산출물 머리에 명시 (예: "잠금 시도자: 에이전트 — 사용자 명시 위임")
   - 위임은 인터뷰 생략이 아니라 **시점 이동** — 인터뷰는 사후 승인 라운드로 미뤄질 뿐, 사라지지 않는다

   **자율 모드 (Autonomous Mode)** — 사용자 발화가 *시작 trigger* 이후 더 이상 없는 경우 (처음부터 끝까지 에이전트 자율). 위임 모드의 극단 형태. 사람 검증 자리가 비기 때문에 *외부 검증을 에이전트 간 합의로 대체*한다:
   - 메인 에이전트가 1차 draft → **서브에이전트가 비판/검증** → 둘이 합의한 형태로 잠금
   - 서브에이전트 호출 시 *적대적 비판* 자리를 의도적으로 박는다 ("이건 제품 아니다" / "약속 휘발했다" / "모방 카테고리다" 같은 가설 시도)
   - 사후 승인 큐는 *다음 메인-서브 합의 시점*에 처리 (사람 승인 자리 없음)
   - 두 LLM이 같은 방향으로 수렴할 위험을 자가 인지 — 서브에이전트 prompt에 *다른 도메인 어휘·다른 관점*을 명시 요청
   - **베이스 밖 어휘 강제** — 서브에이전트 비판 prompt에 *베이스 SKILL 어휘*(약속·자산·verdict·acceptance check 등)만 쓰지 않게 강제. *시각 디자인·콘텐츠 양·UX 동선·시장·사용자 행동·실데이터* 중 1개 이상 어휘를 비판 자리에 박는다. 이것 없이는 *베이스 폐쇄* — 메인이 베이스 어휘 안에서 정당화하면 서브도 베이스 어휘 안에서 비판해 *서비스가 아닌 명세*만 자란다.
   - **페르소나 시뮬레이션은 베이스를 모르는 일반 사용자로** — 외부 사용자 검증을 LLM 페르소나로 대체할 때, 페르소나는 *베이스 어휘를 모르는* 것으로 가정한다. 페르소나가 "promise·acceptance check·자산 trunk"를 자기 발화에 쓰면 *메인 의도와 수렴* — 검증 자리가 비어버린다. 페르소나는 *제품의 외부 표면*(UI·콘텐츠·재방문 동기)만으로 비판해야 한다.
   - 자율 모드 사용 사실을 잠금 산출물 머리에 명시 ("잠금자: 메인-서브 합의 — 자율 모드")

4. **draft 작성** — 인터뷰로 얻은 답(또는 위임 모드의 일방 결정)만 가지고 잠금 산출물 초안을 만든다.
   - 빈 자리를 추측으로 채우지 않는다 — 빈 자리는 빈 자리로 명시
   - 산출물 형식은 도메인이 결정 (베이스는 형식을 강제하지 않는다)

5. **approve 루프** — 사람의 승인을 받기 전에는 잠금이 아니다.
   - 승인되지 않은 항목은 명시적으로 "미승인"으로 표기
   - 사용자가 짧은 승인(`이건 승인` / `이건 빼` / `보류`)을 주면 그대로 처리

   **미승인 표기 형식**:
   - 산출물 끝에 별도 "미승인·빈틈" 섹션 — 잠정 결정의 본 자리
   - 각 항목: 번호 + 항목명 + 잠정 결정 + 사유 + 사후 승인 trigger
   - 본문에는 잠정 결정을 새로 적지 않는다 (잠정값은 별도 섹션에만). 본문 표기 형식(인라인 마킹 등)은 도메인 결정.

6. **잠금 완료 시 Refinement Loop 입력 회수** — 이 인터뷰에서 발견된 의도 빈틈·거부 사유·미해결 항목은 *작동하는 산출물*(SKILL 갱신·계약 변경·fix_plan 항목·코드 TODO 등)로 회수되어야 한다. 별도 텍스트 ledger에 누적만 하면 *문서 더미*. 회수 안 된 항목은 사후 승인 큐(`fix_plan.md` 등)에 *실행 가능한 task* 형태로 등재.

## Refuse First Mode (모드 — 절차 곳곳에 흐른다)

- 진입 시: "잠그지 않을 사유"부터 검토 (Workflow 1)
- 인터뷰 시: 거부 신호 질문 포함 (Workflow 3)
- 잠금 시: 빈 자리를 추측으로 채우지 않음 (Workflow 4)
- 승인 시: 미승인은 미승인 그대로 (Workflow 5)

## Output

- 잠금 산출물 초안 (도메인 형식)
- 미해결·빈틈 목록 (Refinement Loop 입력 후보)
- 명시적 승인 기록
- 잠긴 의도가 이어질 **후속 Gap Ledger 위치**와 **Mechanical Verdict 방식 후보** 명시 (형식은 도메인이 정함; 베이스는 연결점만 잠근다)
- **코드 도달 단위 한 줄** (Workflow 3의 코드 도달 단위 질문 답) — 의도가 다음에 어느 코드·표면·데이터로 떨어지는지 명시. Spiral Loop 운영 중이면 *몇 사이클 안에 도달*까지 함께.
- **제품 도메인 Asset Map 위치** — 새 제품/Spiral Loop라면 각 capability의 `asset_path`와 이번 사이클 기여 위치 명시.

## Boundaries

- **Never**
  - 사용자 승인 없이 산출물을 "최종 잠금"으로 표시
  - 위임 모드라도 사후 승인 큐 등록 없이 일방 결정 (시점 이동 ≠ 인터뷰 생략)
  - 인터뷰 없이 LLM이 자기 추론으로 의도를 채움 (위임 모드는 명시적 발화가 있을 때만)
  - 한 라운드에 4개 이상 질문 던지기
  - 도메인 specific 산출물 형식을 베이스에서 강제 (PRODUCT, Story Chain 같은 이름은 도메인의 일)
  - 미승인 항목을 본문에 섞기 (미승인은 별도 섹션)
- **Ask first**
  - 의도가 4원리와 충돌하는지 불명확할 때
  - 인터뷰 도중 사용자가 잠금 산출물 형식을 바꾸려 할 때
- **Always**
  - **Skill Load Receipt 먼저 남김** — receipt 없이는 다음 단계 진행 금지
  - 단계(discover/draft/approve) 명시
  - "거부할 사유" 검토를 진입 절차로 포함
  - 잠금 완료 시 Refinement Loop 입력 ledger 갱신
  - **코드 도달 단위 한 줄 명시** — Workflow 3의 코드 도달 단위 질문 답. 답할 수 없으면 의도가 너무 추상 → 좁히거나 폐기. Spiral Loop 운영 중이면 *N 사이클 안 도달* 시한도 명시.
  - **자산 기여 명시 (WHO·WHY 포함)** — Workflow 3의 자산 기여 질문 답. 사이클 *시작 시* 박는다 (사후 정당화 금지). WHO·WHY 한 줄 갱신 없으면 의도가 *기능 차원에만 머물러* 제품 지식 휘발 → 사이클 의도 재정의 또는 폐기.
  - **제품 도메인 Asset Map 확인** — 새 제품/Spiral Loop는 Asset Map capability가 존재하거나 명시 거부되어야 한다. `asset_path` 없는 capability는 사이클 시작 차단.
  - **제품별 스킬 자산 확인** — 반복 제품 개발에서 발견된 실행 지식은 문서 설명으로만 두지 말고, 도메인 스킬/플레이북 또는 기계 gate로 회수한다. 스킬이 필요한데 `when_to_use`와 receipt 규칙이 없으면 다음 사이클 시작 차단.
  - **제품 계약 발전 명시** — Workflow 3의 제품 계약 발전 질문 답. 사이클 시작 시 박는다. *사용자 약속의 추가·수정·검증·반증* 중 한 형태로 한 줄. 답 없으면 사이클 진행 차단 — *세 번 폐기 공통 함정(약속 휘발) 차단 자리*.

## Refinement Loop와의 연결

- **입력 제공**: 인터뷰에서 식별된 의도 빈틈, 거부 사유, 미해결 항목 → Refinement Loop의 입력
- **출력 수신**: Refinement Loop가 갱신한 인스턴스화 가이드 (예: "이 도메인에서는 잠금 산출물에 X 항목이 추가로 필요")를 다음 인터뷰에 반영
