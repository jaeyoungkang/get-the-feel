# Korean Prose Skill — Light House 주요 문서용

이 스킬은 Light House에서 *주요 한국어 문서*를 작성하거나 다듬을 때 따르는
산문 규칙을 모은다. 적용 대상은 다음과 같다.

- `docs/*.md` (handoff, design 노트, 기술 가이드, 정책 문서)
- 새로 만드는 top-level README, 운영 문서
- PR description 본문, commit body의 한국어 산문 부분
- Story Chain Human-authority prose (Experience / Moment / Promise / Aspect
  정본 문서). 이때는
  `shared-skills/mission-control/references/human-authority-prose.md`도 함께
  읽고 적용한다.

적용 _대상이 아닌_ 문서:

- Evidence Ledger의 표·코드 블록 본문 — 형식이 정해져 있다.
- LLM judge prompt (`scripts/mission-control/judges/*.prompt.md`) — judge
  품질을 위한 별도 영문 prompt 규칙이 있다.

## 핵심 — 직접 서술

산문은 *제품·시스템이 어떻게 작동하는지를 직접 서술*한다. 모호한 비유나
가상 인용으로 동작을 가리지 않는다.

### 모호한 메타포를 쓰지 않는다

abstract한 비유 표현은 처음 한 번만 정의 용도로 허용된다. 그 이후의 본문은
구체 어휘로 풀어 쓴다.

- ❌ "도장찍기로 통과되지 않는다"
- ✅ "review verdict가 met이지만 실제 근거가 빈 상태로 통과되지 않는다"

- ❌ "코드 본문의 거짓말"
- ✅ "vitest는 통과하지만 실제 invariant를 보호하지 않는 본문" (또는 그
  본문이 무엇을 못 잡는지 구체 서술)

- ❌ "마지막 1cm를 잡는다"
- ✅ "deterministic 게이트가 못 잡는 영역을 결정적으로 잡는다"

비유가 정말 필요하면 *처음 등장 시점*에 풀이를 같이 적는다. 같은 문서에서
다시 쓸 때는 이미 정의된 어휘처럼 다룬다. 같은 비유를 여러 변형으로
재사용하지 않는다.

### 가상 인용을 쓰지 않는다

따옴표로 감싼 인물의 가상 발화는 산문에 넣지 않는다. 운영자/사용자가 무엇을
하는지를 직접 서술한다.

- ❌ "운영자가 '아 이 AC 의미 바뀌었구나'라고 판단하고 review를 추가한다"
- ✅ "운영자는 chip을 보고 AC 의미가 바뀌었음을 인지한다. 새 review entry를
  yaml schema로 작성한다"

### 사용자를 익명으로 표기한다

개별 사용자, 베타 테스터, 리뷰어, 사내 동료의 실명·GitHub handle을 산문에
적지 않는다. "베타 테스터 / 한 베타 테스터 / 동규님 / Donggyu" 같은
식별 표현도 같은 부류로 본다. 의견의 출처를 적어야 하면 "사용자들 의견" /
"user feedback"로 일반화하거나, 중립 식별자(PR 번호, issue 번호, plan
경로)로 가리킨다. 정본 문서(`docs/contracts/**`), Story Chain prose,
admin/decision-log entry, /about/changes entry, commit message,
스킬 reference docs 모두 같은 규칙을 따른다. entry id / i18n key
prefix 같은 슬러그도 마찬가지로 내용 기반 표현을 쓴다
(예: `2026-05-15-search-and-gap-feedback`).

### 명사구 끝맺음을 피한다

영어 번역체 명사구로 마무리하지 않는다.

- ❌ "사용자에게 남아야 할 것은 ... 라는 감각이다"
- ❌ "... 라는 결이다"
- ❌ "이 패턴의 특징은 ... 이다"
- ✅ 그 자리에 어떤 동작·약속이 있는지를 동사로 서술

## 문장 형태

- **단문 중심**으로 쓴다. 한 문장에 한 동작 또는 한 사실. 복문이 길어지면
  끊는다.
- 명사구 도입부("X하는 순간.", "Y가 마주한 순간." 같은 영어 도입체)를
  쓰지 않는다.
- 콜론·세미콜론 남용을 줄인다. 한국어에서는 마침표가 더 자연스럽다.
- "~할 때" / "~하는 경우" 둘이 다 쓸 수 있을 때는 "~할 때"를 우선한다.
  "~인 경우" 도 "~일 때"로 풀 수 있는 자리에서는 풀어 쓴다.

## 어휘

### 영어 단어 인라인

기술 문서에서는 영어 식별자(파일 경로, 함수명, 타입명, 라이브러리 이름,
환경 변수)를 본문에 _직접_ 넣어도 된다. 번역어를 강제로 만들지 않는다. 단:

- 한국어 문장 *형용사 자리*에 영어 단어를 끼우지 않는다.
  - ❌ "*scope*가 좁다"
  - ✅ "범위가 좁다" 또는 "scope가 좁다" (괄호 없이 단독 사용)
- 영어 단어를 *italic emphasis*로 강조 표시하는 것은 한 단락에 한두 번까지.
  강조가 늘어나면 의미가 죽는다.

### 비유 어휘

- 정본에 이미 쓰인 비유 표현(예: 정책 문서의 "단단한 통과 vs 의심스러운
  통과")은 그대로 살린다. 그 위에 새 비유를 쌓지 않는다.
- 새 비유를 만들 때는 그 비유가 가리키는 *구체 어휘*를 같은 문장이나 다음
  문장에 명시한다.

### 정중한 문체와 직설 문체

- 기본은 직설 문체("~한다", "~이다"). 한국어에서 자연스럽게 정중한 문장은
  쓸 수 있지만, 격식체("~하시오")로 강제하지 않는다.
- 사용자에게 직접 행동을 요구할 때는 명령형 또는 중성 권고형 둘 다 OK
  ("~한다" / "~해야 한다" / "~할 것"). 같은 문서 안에서 톤을 일관되게
  유지한다.

## 구조

### 짧은 단락 + 적당한 표

- 단락은 3~5 문장 안쪽으로 끝낸다. 그 이상이면 표나 bullet으로 쪼갠다.
- 표는 카드 형태 데이터에 쓴다. 산문 흐름을 끊지 않게 표 위·아래에 1~2
  문장의 framing을 둔다.
- bullet은 같은 카테고리 항목 3개 이상부터 쓴다. 2개면 산문이 더 자연스럽다.

### 코드 블록과 산문의 분리

- 코드 블록은 _그 자체로_ 사례 역할을 한다. 코드 블록 위 산문은 무엇을
  보여주는지 한 줄로 frame한다. 같은 정보를 코드 블록 아래에서 다시 풀어
  설명하지 않는다.
- 코드 블록 안의 주석으로 설명을 보내는 것이 산문보다 명확할 때가 많다.

### 섹션 헤더

- "왜 필요한가", "사용법", "사례", "검증" 같은 짧은 섹션 헤더는 가이드 문서에
  자연스럽다.
- 헤더에 의문문("이 게이트가 정말 필요한가?")을 쓰지 않는다. 본문이
  자문자답하는 형태가 된다.

## 작성 흐름

- 문서 단위로 draft를 한 번 쓴 뒤 _위 규칙으로 한 번 훑어 정리한다_. 처음
  쓸 때부터 모든 규칙을 지키려고 하지 않는다.
- 비유나 가상 인용은 처음 쓸 때 자연스러워 보인다. 두 번째 검토에서 거의
  대부분 풀어 써지므로, 이 단계를 건너뛰지 않는다.
- 큰 변경(여러 섹션 동시 수정)은 commit 전 한 번 더 단문 중심·번역체
  점검을 한다.

## 사례 — `verification-gates.md`의 비유 정리

2026-05-07 작업한 `docs/verification-gates.md`에서 *도장찍기*와 _거짓말_
비유를 사용해 처음 초안을 썼다. 두 번째 검토에서 본 스킬 규칙으로 정리:

| 초안                                                           | 정리 후                                                     |
| -------------------------------------------------------------- | ----------------------------------------------------------- |
| "사람이 met이라 적었지만 실제로는 미증명인 상태" 정도까지는 OK | 한 번 정의 후 그대로 사용                                   |
| "도장찍기로 통과되지 않게"                                     | "review verdict가 met이지만 근거가 빈 상태로 통과되지 않게" |
| "코드 본문의 거짓말"                                           | "vitest는 통과하지만 invariant를 보호하지 않는 본문"        |
| "마지막 1cm를 잡는다"                                          | "그 영역을 결정적으로 잡는다"                               |

이 사례를 새 문서에서도 참고한다. 같은 비유를 다시 쓰지 않는다.

## 다른 스킬과의 관계

- **`shared-skills/mission-control/references/human-authority-prose.md`** —
  Story Chain Human-authority prose 전용. Human-authority prose를 작성하거나
  다듬을 때는 본 스킬을 먼저 켜고 이 reference도 함께 읽는다. 충돌하면 더
  좁고 엄격한 이 reference가 우선한다.
- **`shared-skills/about-prose/SKILL.md`** — `/about`, `/admin/intent`의
  commitment prose 전용. 사용자에게 공개되는 카피 톤 규칙. 본 스킬과 별개
  layer.

본 스킬은 일반 docs 한국어 산문을 담당한다. Story Chain Human-authority
prose에서는 공통 한국어 산문 기준을 제공하고, 세부 기준은
`human-authority-prose.md`가 좁힌다.
