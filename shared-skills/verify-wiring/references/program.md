# verify-wiring — 배선 검증 레퍼런스

구현 후 "돌아간다"에서 "작동한다"까지 가는 검증 절차다.

---

## 배경

W11~W12 PAR Loop 이식에서 발견한 패턴: 스키마나 타입을 추가하고 `tsc`가 통과해도 실제로는 앞단(생성)만 구현하고 뒷단(소비)을 안 뚫은 경우가 반복됐다.

대표 실패 패턴:

- 생성은 됐지만 execute에 주입되지 않음
- 계산은 됐지만 실제 슬롯/행동에 반영되지 않음
- audit 결과를 저장만 하고 다음 턴 plan에 되먹이지 않음
- 필드는 정의됐지만 갱신 로직이 없음
- 도구 실행 결과는 있지만 audit나 의사결정으로 연결되지 않음
- 타입은 생겼지만 어댑터에서 값을 채우지 않음

---

## 실행 절차

### 입력

- 대상 저장소 경로
- 검증할 변경 범위
  - 기본: `git diff HEAD`
  - 필요하면 경로, 모듈, 커밋 범위로 축소

### Step 1: 변경 추출

1. diff에서 새로 추가된 필드, 타입, 파라미터, 상태 키, 프롬프트 반환 필드를 추출한다.
2. 각 항목마다 "어디서 생성되는가?"를 먼저 기록한다.

### Step 2: 소비 경로 추적

각 항목마다 다음을 확인한다.

1. 프로젝트 전체에서 해당 항목을 읽는 코드를 검색한다.
2. 읽는 코드가 없으면 `missing-consumer`다.
3. 읽는 코드가 있어도 로깅, 저장, 단순 전달뿐이면 `logging-only`다.
4. 실제 분기, 렌더링, 계획, 실행, 검증, 상태 전이 중 하나에 영향을 줘야 `connected`로 본다.

### Step 3: 프롬프트 계약 검증

LLM이 반환해야 하는 필드라면:

1. 프롬프트 지시에 필드가 명시돼 있는지 확인한다.
2. 예시, 스키마, 구조화 출력 정의에도 필드가 반영돼 있는지 확인한다.
3. 빠져 있으면 `prompt-contract-missing`이다.

### Step 4: 피드백 루프 검증

순환 구조가 있다면:

1. A의 출력이 B로 전달되는지 확인한다.
2. B의 출력이 다시 A 또는 다음 턴의 plan/control로 돌아오는지 확인한다.
3. 한쪽이라도 끊기면 `feedback-loop-broken`이다.

### Step 5: 리포트 생성

다음 형식으로 정리한다.

| item | generated | consumed | verdict | note |
|------|-----------|----------|---------|------|
| situational_promises | `plan.ts:45` | 없음 | `missing-consumer` | 앞단만 존재 |

---

## 판정 기준

| verdict | 의미 |
|---------|------|
| `connected` | 생성에서 소비까지 의미 있는 경로가 확인됨 |
| `logging-only` | 읽기는 하지만 행동 변화로 이어지지 않음 |
| `missing-consumer` | 생성은 있지만 소비 경로가 없음 |
| `prompt-contract-missing` | LLM 반환 필드가 프롬프트 계약에 없음 |
| `feedback-loop-broken` | 순환 경로의 한쪽이 끊김 |

---

## 점검 팁

- 시작점은 diff, 끝점은 행동 변화다.
- "검색에 잡힌다"와 "실제로 쓰인다"를 구분한다.
- 테스트 코드만 참조하는 경우는 보조 근거일 뿐, 런타임 연결 증거로 단정하지 않는다.
- 이름이 바뀌는 어댑터 구간은 매핑을 먼저 확인한다.
